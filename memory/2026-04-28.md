# Mémoire session — 2026-04-28

## Module RSEvents Pro — Upcoming Events (homepage)

**Problème :** Le module affichait le 24 avril 2026 (passé) parmi les 4 prochains événements.

**Cause identifiée :** Le paramètre `events` dans `bwhwo_modules` avait la valeur `8`, non gérée par le code RSEvents Pro → aucun filtre de date appliqué.

**Correctifs appliqués :**
1. Paramètre `events` corrigé en `2` (upcoming) dans la table `bwhwo_modules`
2. Filtre UTC-safe ajouté directement dans le template override `mod_rseventspro_upcoming` : `strtotime($event->start . ' UTC') < time()` → skip
3. Cache purgé (×2)
4. Résultat : les 4 événements suivants s'affichent correctement en FR, ZH et EN (8 mai, 22 mai, 2 juin, 9 juin 2026)

**Fichier override :** `/tmp/mod_rse_upcoming_override.php` (source locale), déployé via SFTP sur le serveur

## Analyse extensions RSEvents Pro / RSForm Pro

- **RSEvents Pro :** fonctionnel mais fragile hors cas standards. Paramètres de module peu documentés, timezone ambiguë dans certains helpers. Overrides en place compensent bien.
- **RSForm Pro :** correct pour usage défini. Logique métier dispersée dans des champs de config (JS, ScriptBeforeDisplay). Ne pas réécrire maintenant — rapport effort/bénéfice défavorable.
- **Plugin PayPal RSEvents :** IPN déprécié depuis 2020. Validation montant fragile (`amount` lu depuis `$_POST` avant vérification). Mode sandbox silencieusement dangereux. Non utilisé sur le site.
- **Plugin Stripe RSEvents :** 4 modes dont 3 obsolètes/non conformes PCI/PSD2. Mode 3 (Checkout Sessions) seul valide mais webhook sans vérification de signature. Ne pas activer en l'état.
